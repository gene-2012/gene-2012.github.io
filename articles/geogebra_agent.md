## 0x01 引言

在学习数学（特别是做新定义类题目）时，往往需要大量绘图。然而，传统绘图的一大缺陷是无法体现“动点”、“动角”以及变量取值等。因而产生了 Geogebra 这类数学绘图工具，极大简化了数学绘图过程。

不过，绘制 Geogebra 图像时，也会耗费一定时间，容易打断思路。假如设计一个自动化工具，将题目（甚至是题目图片）提供给 LLM，并由其快速生成图像，这一情况是否能够改善？

## 0x02 方案计划

面对这一问题，考虑两种可能的解决方案：
1. 单次对话，由 AI 输出一个结构化的 JSON，包含对 Geogebra 文件的描述（这很类似 MathMover 平台的思路）。
2. 多轮对话，向 AI 暴露若干 Function-Call 工具，每次对话调用某一工具以实现几何图形的绘制和几何约束的描述。

上述两种方法均使用 [`deployggb.js`](https://geogebra.github.io/docs/reference/en/GeoGebra_Apps_Embedding/) 进行渲染。出于使用便利性和稳定性的考量，我们最终选择了**方案 2**。

## 0x03 核心逻辑：定义工具集

在上述代码中，我们通过封装 `window.ggbApplet` 的 API，构建了一套可以直接映射到 LLM **Function-Call** 的工具链。其核心在于将复杂的几何约束（如“过 A 点作 BC 的平行线”）简化为符合 GeoGebra 语法的 `definition` 字符串。

### 1. 核心绘图工具 (GeoGebra Core)
这些工具直接操作画布，是 Agent 绘图的核心。

| 工具名称 | 功能描述 | 关键参数 |
| :--- | :--- | :--- |
| `add_element` | **添加几何元素**。支持点、线、圆、函数、轨迹等。 | `id`, `definition` (GGB语法), `color`, `label`, `trace` |
| `add_slider` | **创建滑动条**。用于控制动点、角度或数值变量。 | `id`, `min`, `max`, `step`, `value` |
| `remove_element` | **删除对象**。根据 ID 移除指定的几何元素。 | `id` |
| `list_elements` | **查看对象列表**。返回当前画布上所有已存在的元素信息。 | 无 |
| `clear_canvas` | **清空画布**。重置所有设置并删除所有元素。 | 无 |
| `set_canvas` | **调整视窗**。设置坐标轴的显示范围（缩放/位移）。 | `xmin`, `xmax`, `ymin`, `ymax` |
| `start_animation` | **启动动画**。让所有被设为“动画激活”的对象动起来。 | 无 |

### 2. 任务管理工具 (Task Management)
用于 Agent 在多步绘图逻辑中记录思路和待办事项。

| 工具名称 | 功能描述 | 关键参数 |
| :--- | :--- | :--- |
| `todo_add` | 添加一个绘图步骤或逻辑待办。 | `content`, `priority` |
| `todo_list` | 列出当前所有未完成的任务，防止 Agent 遗忘步骤。 | 无 |
| `todo_done` | 将某个绘图任务标记为已完成。 | `todo_id` |
| `todo_delete` | 删除一个任务项。 | `todo_id` |

## 0x04 实测

考虑到任务足够简单，使用 Javascript 实现了一个简单的 Agent 循环，使之可在浏览器单页面中正确运行。下面是一个例子（可互动 Demo 将于近日上线）。

> 在平面直角坐标系 $xOy$ 中，函数 $y = kx + b$ ($k \neq 0$) 与 $y = x - k$ 的图象交于点 $(2, 0)$。
> 
> (1) 求 $k$，$b$ 的值；
> 
> (2) 当 $x \le 0$ 时，对于 $x$ 的每一个值，函数 $y = mx + n$ 的值大于函数 $y = kx + b$ ($k \neq 0$) 的值，且小于函数 $y = x - k$ 的值，直接写出 $m$ 和 $n$ 的取值范围。

![Demo Image](/articles/img/geogebra_01.png)

可以看到，LLM 不仅正确的绘制出了图像，还顺便正确解出了第二问的答案。

## 0x05 深度观察：Agent 如何思考几何？

在实测过程中，我们发现方案 2（多轮对话 + Function-Call）展现出了意想不到的“联想”能力。相比于传统的静态解析，Agent 的绘图逻辑遵循以下路径：

1. **约束识别**：Agent 首先识别出已知条件中的“硬约束”（如点 $(2, 0)$）。它会先调用 `add_element` 固定该点，以此为锚点构建后续图像。
2. **动态探究**：在处理第 (2) 问关于 $m, n$ 的范围时，Agent 并未直接进行代数推导，而是先调用 `add_slider` 创建了代表 $m$ 和 $n$ 的滑动条，并绘制出直线 $L: y = mx + n$。

## 0x06 方案对比与优势

| 维度 | 方案 1：JSON 一次性生成 | 方案 2：Function-Call 迭代 (本文) |
| :--- | :--- | :--- |
| **容错性** | 极低。语法错误会导致整个渲染失败。 | **高**。单步出错可由 Agent 自行纠错重试。 |
| **逻辑深度** | 仅限于简单的图形还原。 | **深**。支持通过滑动条探索动态几何边界。 |
| **交互体验** | 静态。 | **动态**。用户可实时看到绘图过程，甚至介入引导。 |

## 0x07 展望

未来，或许可以依赖于多模态大模型，提供一个截图工具，可以传入当前 Geogebra 的图片，以达到验证图形的目的。以及，或许真的可以通过这样一套方法，让大模型观察图片，寻找解题思路并最终构建一个足够强大的解题 Agent。
